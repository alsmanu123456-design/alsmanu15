# WhatsApp Bot Pro v8.0 — دستور المشروع

> **هذه القواعد غير قابلة للتفاوض.**
> أي Agent أو مطور يعمل على هذا المشروع مُلزَم بها.
> في حالة التضارب بين هذه القواعد وأي طلب، القواعد مُقدَّمة.
> آخر تحديث: 2026-06-28

---

## القسم الأول — قواعد الملفات والبنية

### Q1.1 — ممنوع إنشاء ملفات بأسماء مرفوضة

**ممنوع** إنشاء أي ملف يبدأ بـ:
- `patch-*` — استخدم الوحدة الصحيحة
- `fix-*` — استخدم الوحدة الصحيحة
- `test-*` — استخدم `tests/` بدلاً منها
- `final-*`, `v2-*`, `new-*`, `temp-*`, `tmp-*`

**ماذا تفعل بدلاً من ذلك:**
```
❌ إنشاء patch-reconnect-fix.mjs
✅ تعديل dist/session-engine.mjs مباشرة
```

---

### Q1.2 — ممنوع التعديل المباشر على dist/index.mjs

`dist/index.mjs` = ملف مُجمَّع (esbuild output).
**ممنوع** إجراء أي تعديل يدوي عليه إلا عبر:
- الآلية الرسمية لـ `infrastructure/patch-manager.mjs`
- أو بناء من المصدر TypeScript

**الاستثناء الوحيد:** تعديلات طارئة (hotfix) حرجة وموثّقة في REFACTOR_PLAN.md.

---

### Q1.3 — كل ملف مسؤولية واحدة فقط (SRP)

إذا استطعت وصف ملفك بـ "يفعل X **و** Y"، فهو يحتاج للتقسيم.

**معايير الحجم المقبول:**
| الطبقة | الحد الأقصى المقبول |
|---|---|
| utils/ | 100 سطر |
| core/ | 150 سطر |
| infrastructure/ | 200 سطر |
| bootstrap/ | 60 سطر |
| dist/*.mjs (module) | 500 سطر |

**تجاوز الحد لا يمنع العمل، لكنه مؤشر على أن الملف يحتاج تقسيم.**

---

### Q1.4 — هيكل المجلدات ثابت

```
alsmanu5/
├── bootstrap/     ← تسلسل التشغيل فقط
```

**ممنوع:** إنشاء مجلدات جديدة خارج هذا الهيكل دون توثيق في ARCHITECTURE.md.

## القسم الثاني — قواعد الكود

### Q2.1 — لا نسخ للمنطق (DRY)

إذا كتبت نفس الكود في مكانين، أنشئ وحدة مشتركة.

```
❌ كتابة decode() في config.mjs وأيضاً في deploy.mjs
✅ استخدام utils/token-codec.mjs من كلاهما
```

---

### Q2.2 — لا استيراد مباشر بين Modules

Module لا يستورد Module آخر مباشرة.
التواصل يكون عبر:
- **Services** المشتركة
- **Events** (emit/on)
- **dist/index.mjs** كوسيط

```
❌ ai.mjs يستورد من points.mjs
✅ ai.mjs يستدعي deductPoints() من database (shared)
```

---

### Q2.3 — Logger واحد فقط

المصدر الوحيد للطباعة في طبقة Bootstrap/Core/Infrastructure:
```
core/logger.mjs → ok(), wrn(), inf(), err()
```

في طبقة Engine (dist/):
```
src/lib/logger.ts → pino logger
```

**ممنوع:** استخدام `console.log()` في أي ملف من طبقات Bootstrap/Core/Infrastructure.

---

### Q2.4 — الأخطاء لا توقف البوت

```javascript
// ✅ صحيح
try {
  await processFeature(data);
} catch (e) {
  logger.warn({ err: e.message }, "Feature failed, continuing");
}

// ❌ خاطئ
await processFeature(data); // إذا فشل → البوت يموت
```

**الاستثناء:** أخطاء Health Check في bootstrap → `process.exit(1)` مقبول (يمنع تشغيل بوت مكسور).

---

### Q2.5 — العمليات الثقيلة في Queue

أي عملية تستغرق > 5 ثوانٍ أو تستهلك > 100MB يجب أن:
1. تعمل في خلفية (async)
2. تُرسل إشعار بالبدء للمستخدم
3. تُرسل إشعار بالانتهاء/الفشل

```javascript
// ✅ صحيح
await bot.sendMessage(chatId, "⏳ جارٍ التحميل...");
const result = await downloadVideo(url); // عملية ثقيلة
await bot.sendMessage(chatId, "✅ تم التحميل");

// ❌ خاطئ
const result = await downloadVideo(url); // المستخدم ينتظر بدون إشعار
```

---

### Q2.6 — العمليات الدورية عبر Scheduler موثّق

كل `setInterval()` أو `setTimeout()` يجب أن:
1. يكون موثّقاً في ARCHITECTURE.md (قسم Background Tasks)
2. يكون له اسم واضح (لا anonymous functions)
3. يكون قابلاً للإيقاف (احتفظ بـ reference)

```javascript
// ✅ صحيح
const hourlyReconnect = setInterval(runHourlyReconnect, 60 * 60 * 1000);

// ❌ خاطئ
setInterval(async () => { /* ... */ }, 60 * 60 * 1000);
```

## القسم الثالث — قواعد البيانات والبيانات

### Q3.1 — لا حذف لبيانات المستخدمين

```
❌ deleteUser(userId) → يحذف فعلياً
✅ deactivateUser(userId) → يضع status: "deleted"
```

البيانات التاريخية دائماً مهمة. الحذف الصعب (hard delete) ممنوع.

---

### Q3.2 — التوافق مع البيانات القديمة

أي تغيير في schema يجب أن:
1. يكون backward compatible
2. يستخدم `data?.newField ?? defaultValue`
3. لا يكسر قراءة JSON القديم

```javascript
// ✅ صحيح
const limit = user.newLimit ?? user.oldLimit ?? 5;

// ❌ خاطئ
const limit = user.newLimit; // يكسر المستخدمين القدامى الذين ليس لديهم newLimit
```

---

### Q3.3 — حفظ تلقائي دائم

أي تعديل على `inMemoryDB` يجب أن:
- يُحفظ في `bot-data/*.json` خلال 3 دقائق كحد أقصى
- **أو** يُستدعى `saveDatabase()` صراحةً عند العمليات الحرجة

## القسم الرابع — قواعد الأداء والقابلية للتوسع

### Q4.1 — الهدف: 10,000 مستخدم

كل قرار تقني يُقيَّم بمعيار: "هل يعمل مع 10,000 مستخدم؟"

```
✅ LRU Cache مع حد أقصى → يعمل مع 10,000
❌ Map بلا حد → RAM يستنزف مع 10,000
```

---

### Q4.2 — لا عمليات متزامنة ثقيلة

```javascript
// ❌ خاطئ — يُجمّد event loop
for (const user of allUsers) {
  sendMessage(user.id, msg); // بدون await + بدون delay
}

// ✅ صحيح
for (const user of allUsers) {
  await sendMessage(user.id, msg);
  await sleep(100); // rate limiting
}
```

---

### Q4.3 — Rate Limiting على كل واجهة

- واجهة تيليجرام: لا أكثر من 30 رسالة/ثانية للبوت
- واجهة واتسآب: لا أكثر من رسالة كل 2 ثانية لنفس الجهة
- واجهة OpenAI: احتفظ بـ token budget لكل مستخدم

## القسم السابع — قواعد إعادة الهيكلة

### Q7.1 — أي Refactor يحافظ على السلوك الحالي

**قبل** أي refactor:
1. وثّق السلوك الحالي (inputs/outputs متوقعة)
2. نفّذ الاستخراج
3. تحقق أن السلوك لم يتغير

---

### Q7.2 — اختبار بعد كل استخراج

```
[استخراج وحدة] → [تشغيل البوت] → [مراجعة السجلات] → [اختبار يدوي]
```

لا تنتقل لاستخراج التالي إلا بعد تأكد عمل السابق.

---

### Q7.3 — لا refactor وإصلاح وميزة في نفس الوقت

كل commit يجب أن يكون **إما:**
- A) refactor (تغيير هيكلي، لا تغيير وظيفي)
- B) bugfix (إصلاح خطأ، لا تغيير هيكلي)
- C) feature (ميزة جديدة، لا تغيير هيكلي)

خلط الثلاثة في commit واحد = مشكلة مستقبلية.

## القسم الثامن — سلم الأولويات

عند التضارب بين متطلبين، اتبع هذا الترتيب:

```
1. 🔴 الاستقرار (البوت يعمل)
2. 🟠 أمان البيانات (لا فقدان)
3. 🟡 الأمان (لا اختراق)
4. 🟢 الأداء (سرعة الاستجابة)
5. 🔵 الميزات الجديدة
6. ⚪ جماليات الكود
```

**مثال:** إذا كان refactor قد يكسر استقراراً → أجّله وأصلح المشكلة أولاً.
